# 发现: 元数据标准与舰长批注规范

**日期**: 2026-04-15  
**类型**: 规范

---

## 摘要

为了让母舰笔记具备「可追溯、可筛选、可进化」的能力，设计一套宽松的 frontmatter 元数据标准，以及两种舰长批注形式（inline + 独立文件）。

---

## 1. 元数据标准（Frontmatter）

### 1.1 强制字段（知识类笔记）

所有属于「学科知识」的 Markdown 文件，**必须**包含以下两个字段：

```yaml
---
generation: 1
subject: Math
---
```

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `generation` | int | 创建该笔记的舰长代际 | `1`, `2`, `3` |
| `subject` | string | 所属学科/领域 | `Math`, `专业课`, `English`, `政治`, `方法论` |

### 1.2 推荐字段（可选但鼓励）

```yaml
---
generation: 1
captain: "第一代舰长"
subject: Math
stage: "强化阶段"
chapter: "第 2 讲 - 数列极限"
tags: ["数列极限", "递推法", "单调有界"]
source: "张宇强化18讲 + 个人整理"
date-created: 2025-07-15
date-modified: 2025-08-03
---
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `captain` | string | 创建者署名 |
| `stage` | string | 备考阶段（基础/强化/真题/冲刺） |
| `chapter` | string | 所属章节 |
| `tags` | list | Obsidian 标签，便于检索 |
| `source` | string | 知识来源（老师/书籍/视频） |
| `date-created` | date | 创建日期 |
| `date-modified` | date | 最后修改日期 |

### 1.3 豁免文件（不需要 frontmatter）

以下类型的文件可以不写 frontmatter：
- 纯模板文件（`Templates/` 目录下）
- 日记、随笔、个人情绪记录（这些本就不应进入母舰）
- `README.md`, `CONTRIBUTING.md` 等仓库说明文档
- 简短的清单/看板文件（如 `xxx任务看板.md`）

### 1.4 代际更新规则

当第二代舰长修改第一代笔记时，**不修改原文件的 `generation`**。原文件代表「原始知识」的代际，舰长批注代表「后续改进」的代际。

如果第二代舰长**重写**了整篇文章，则新建文件并标注 `generation: 2`。

---

## 2. 舰长批注规范

### 2.1 形式 A：Inline 批注（适用于精华补充、小修正）

在原文相关段落的下方，用引用块 + 特定图标标注：

```markdown
数列极限的递推法核心思路是：先证单调有界，再求极限。

> 🚀 **Gen-2 舰长批注** (@学弟A): 
> 这里补充一个快速判断单调性的技巧：如果递推式 $x_{n+1}=f(x_n)$ 且 $f'(x)>0$，则数列单调；若 $f'(x)<0$，则数列振荡。
```

#### Inline 批注规范

- **前缀**: `> 🚀 **Gen-{N} 舰长批注** (@GitHub用户名): `
- **内容长度**: 不超过 3 段。如果超过，改用独立文件。
- **位置**: 紧跟在原文相关段落之后，用空行分隔。
- **禁止**: 直接修改原文（除非是明显的笔误/错别字）。

### 2.2 形式 B：独立文件批注（适用于大量补充、不同观点）

在原文同级目录下，新建一个以 `-批注-gen{N}` 为后缀的文件：

```
原文: 知识枢纽：数列极限.md
批注: 知识枢纽：数列极限-批注-gen2.md
批注: 知识枢纽：数列极限-批注-gen3.md
```

#### 独立文件模板

```markdown
---
generation: 2
captain: "学弟A"
subject: Math
chapter: "第 2 讲 - 数列极限"
annotation-of: "知识枢纽：数列极限.md"
annotation-type: ["补充例题", "方法优化"]
date: 2026-04-15
---

# 知识枢纽：数列极限 —— Gen-2 舰长批注

> 🚀 **批注来源**: [[知识枢纽：数列极限]]
> 🚀 **批注舰长**: @学弟A (Gen-2)
> 🚀 **批注日期**: 2026-04-15

---

## 补充 1: 递推数列的"先斩后奏"法

（此处写大量补充内容...）

## 补充 2: 一道易错题的三种解法

（此处写大量补充内容...）

---

## 与原文观点的差异说明

*如果本批注与原文观点不同，在此说明理由。*
```

#### 独立文件命名规则

```
{原文文件名（不含 .md）}-批注-gen{N}.md

示例:
- 战术卡片：极限保号性-批注-gen2.md
- 2020T2--细节版-批注-gen3.md
```

### 2.3 形式选择决策树

```
你的补充/批注是...
    │
    ├── 字数 < 100 字，且是直接补充/修正原文某句话
    │   └── → 用 Inline 批注
    │
    ├── 字数 > 100 字，或包含例题/图解/多步骤推导
    │   └── → 用 独立文件批注
    │
    ├── 与原文观点不同，但你认为值得记录
    │   └── → 用 独立文件批注（并在开头说明差异）
    │
    └── 大量重写原文，几乎变成新文章
        └── → 新建一篇独立文章，generation = 你的代际
```

## 3. Dataview 查询示例

利用统一的 frontmatter，母舰可以构建强大的查询看板：

### 查询某学科全部 Gen-2 批注

```dataview
TABLE captain, annotation-type, date
FROM ""
WHERE file.name contains "批注-gen2"
SORT date DESC
```

### 查询全部未批注的 Gen-1 知识点

```dataview
TABLE generation, subject, chapter
FROM ""
WHERE generation = 1 AND !file.name contains "批注"
SORT subject ASC
```

### 查询舰长名录（基于批注和文件元数据）

```dataview
TABLE WITHOUT ID captain, count(captain) as "贡献数"
FROM ""
GROUP BY captain
SORT count(captain) DESC
```

## 4. 实施建议

### 对第一代舰长（你）
- 优先给核心知识枢纽文件补充 `generation: 1` + `subject`。
- 不需要一次性给 1000+ 个文件都加，先给「章节汇总」「知识枢纽」「战术卡片」加。
- 日常写新笔记时，顺手带上 frontmatter。

### 对第二代及以后的舰长
- 发 PR 时，如果修改了老文件，检查是否需要在附近加 inline 批注。
- 如果贡献了大量新内容，新建独立文件并标注 `generation: 你的代际`。
- PR 描述里引用你修改/批注的文件名。

---

## 影响

该规范确保了：
1. **可追溯**：每篇笔记和批注都知道是谁、哪一代写的。
2. **可筛选**：Dataview 可以动态生成「Gen-2 新增内容」「某学科全部批注」等看板。
3. **不侵入**：Inline 批注和独立文件都不破坏原文结构，读者可以自由选择是否阅读批注。
4. **低门槛**：只需要两个强制字段，不增加日常笔记负担。
