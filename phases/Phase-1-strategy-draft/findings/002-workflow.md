# 发现: 舰长传承工作流设计

**日期**: 2026-04-15  
**类型**: 架构 / 流程

---

## 摘要

设计一套清晰、可复制的「代际传承工作流」，让每一位新舰长都能理解：如何基于母舰笔记进行 Fork、本地优化、最终通过 Pull Request 反哺社区。

---

## 1. 核心工作流概览

```
┌─────────────────┐
│  第一代舰长      │  创建母舰，推送初始笔记
│  (你，Gen-1)    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  qiusuo-kaoyan  │  Public Org + notes 仓库
│    母舰         │
└────────┬────────┘
         │
    ┌────┴────┐
    │  Fork   │  学弟学妹点击 Fork，获得自己的副本
    ▼         ▼
┌─────────┐ ┌─────────┐
│ 学弟A   │ │ 学妹B   │  在自己的 Fork 中修改、补充
│ (Gen-2) │ │ (Gen-2) │
└────┬────┘ └────┬────┘
     │           │
     ▼           ▼
┌─────────────────────────┐
│     Pull Request        │  将改进提交回母舰
│  "Gen-2 高数极限专题补充" │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│   Review & Merge         │  第一代/资深舰长审核后合并
│      母舰升级            │
└─────────────────────────┘
```

## 2. 新舰长上船指南（简化版）

### Step 1: Fork 母舰
1. 访问 `https://github.com/qiusuo-kaoyan/notes`
2. 点击右上角 **Fork** 按钮
3. 等待几秒钟，获得 `你的用户名/notes` 仓库

### Step 2: 克隆到本地
```bash
git clone https://github.com/你的用户名/notes.git
cd notes
```

### Step 3: 用 Obsidian 打开
1. 打开 Obsidian
2. 选择 **"打开本地仓库"** → 选择 `notes` 文件夹
3. 安装提示的社区插件（Dataview、Templater 等）
4. 开始阅读、批注、补充自己的内容

### Step 4: 提交你的改进
```bash
git add .
git commit -m "gen2: 补充数列极限的递推法示例"
git push origin main
```

### Step 5: 发起 Pull Request
1. 打开 GitHub 网页，进入你的 Fork 仓库
2. 点击 **"Contribute" → "Open pull request"**
3. 填写 PR 标题和说明，参考模板：
   - **标题**: `gen2: 高数-第2讲 数列极限补充`
   - **说明**: 补充了递推法示例，修正了原笔记中的一个笔误。
4. 等待 Review，合并后你的贡献就成为母舰的一部分！

## 3. PR 规范

### 命名规范

```
gen{N}: {学科}/{章节} - {简要描述}

示例:
- gen2: Math/数列极限 - 补充递推法示例
- gen2: 专业课/自控 - 更新第3章简答题
- gen3: English/文化补充 - 新增2026T1解析
```

### PR 描述模板

```markdown
## 舰长信息
- **代际**: Gen-2
- **舰长**: @你的GitHub用户名
- **学科**: Math
- **章节**: 第 2 讲 - 数列极限

## 变更说明
- 补充了 xxx
- 修正了 xxx
- 删除了 xxx

## 批注类型
- [ ] 新增知识点
- [ ] 修正错误
- [ ] 优化表述
- [ ] 补充例题
- [ ] 其他

## 检查清单
- [ ] 已更新 frontmatter 中的 `generation` 字段
- [ ] 未包含个人隐私信息
- [ ] 未上传版权敏感 PDF
```

## 4. Review & Merge 规则

### Reviewer 分配
- 默认由 `Captains` Team 中的第一代/资深舰长 Review。
- 如果 PR 修改量较小（< 10 个文件，无争议），可由任何 Captain 直接合并。
- 如果 PR 涉及大量知识观点变更，需要至少 **2 个 Captain 批准**。

### Merge 策略
- 使用 **"Squash and merge"**，保持 `main` 分支历史整洁。
- Merge 后，PR 标题必须规范，因为它会成为 Release Changelog 的一部分。

---

## 5. 舰长晋升机制

### 如何成为 Contributor
- 向母舰提交过至少 **1 个被合并的 PR**。
- 由现有 Captain 邀请加入 `Contributors` Team。

### 如何成为 Captain (Gen-N)
- 成功上岸（考研录取）。
- 在备考期间向母舰提交过 **3 个以上被合并的 PR**，或主导过一次 **Release** 的整理工作。
- 由现有 Captain 提名，Org Owners 批准后邀请加入 `Captains` Team。

### Captain 职责
1. Review 新舰长的 PR
2. 维护母舰的目录结构和文档
3. 每年考研季后主导一次 Release 整理
4. 培养下一代 Captain

---

## 6. 特殊情况处理

### 两位舰长对同一知识点持不同观点
- **策略**: 允许存在！使用独立文件批注（如 `xxx-批注-gen2.md` 和 `xxx-批注-gen3.md`），不强制合并为单一观点。
- **原则**: 母舰是知识的「星图」，不是唯一真理。每代舰长标注自己的 `generation`，读者自行判断。

### Fork 后长期未 PR，但笔记已面目全非
- **策略**: 这是被鼓励的！即使没有 PR，Fork 本身也是对母舰价值的认可。
- 如果未来某天想反哺，可以 cherry-pick 部分改进发起 PR。

### 用户毕业后不再活跃
- **策略**: 在毕业前将 Org Ownership 移交给第二代 Captain 中的一位核心维护者。
- 母舰的生命力在于「船在人在，船长可换」。

---

## 影响

该工作流确保了：
1. **低门槛**：Fork + Obsidian 打开即可开始，不需要复杂的 Git 知识。
2. **高质量**：PR + Review 机制保证母舰内容的可靠性。
3. **强归属**：Contributor → Captain 的晋升路径提供了持续参与的动力。
4. **可演进**：不同观点可以共存，避免知识僵化。
